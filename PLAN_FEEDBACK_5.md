## Verdict

Revision 3.2 is ready to move into implementation. The architecture, milestone boundaries, persistence model, request pipeline, credential handling, and recovery strategy are now coherent. 

I would make six final specification patches. None requires a major redesign.

## 1. Define whether query rows store encoded or decoded text

The URL model promises to preserve percent-encoding, but it does not define the representation stored in `QueryParam.key` and `value`. 

For example:

```text
?q=hello%20world
?q=hello+world
?q=hello world
```

You need to decide what appears in the parameter editor and what is stored internally. Otherwise, URL-bar edits and parameter-row edits may double-encode values or change their representation unexpectedly.

A lean and lossless approach:

```text
QueryParam.key and value store the encoded substrings exactly as entered.

URL-bar parse:
- split without decoding
- preserve %XX sequences and +
- preserve malformed percent sequences

Parameter-row serialization:
- write stored values verbatim
- do not automatically percent-encode

Optional later feature:
- separate decoded-value editing mode
```

This is less convenient than automatically encoding spaces, but it guarantees round-tripping and avoids ambiguity around templates such as `{{searchTerm}}`.

Alternatively, store decoded text, but then you need a defined encoder, malformed-encoding behavior, and a way to preserve whether spaces were represented as `+` or `%20`.

## 2. Add an explicit `prepareRequest` option matrix

The pipeline always describes resolving variables and substituting fields, while `resolveVariables: false` is used for default cURL output. The exact branching behavior is still implicit. 

Add a table or equivalent contract:

| Consumer                   | Resolve variables | Include sensitive values | Purpose                 |
| -------------------------- | ----------------: | -----------------------: | ----------------------- |
| Send                       |               Yes |                      Yes | Actual request          |
| Preview                    |               Yes |                       No | Safe resolved preview   |
| Copy cURL                  |                No |                       No | Shareable template      |
| Copy cURL with credentials |               Yes |                      Yes | Executable reproduction |
| Future codegen             |      Configurable |            No by default | Safe generated code     |

Also clarify:

* Recursive variable values may still be analyzed for cycles when substitution is disabled.
* Unresolved variables block **Send**, but should not block template cURL generation.
* `maskSecrets` and `redactCredentials` should not produce conflicting outcomes.

A clearer API may be:

```ts
interface PreparationOptions {
  variableMode: "resolve" | "preserve";
  sensitiveValueMode: "include" | "redact";
  unresolvedMode: "error" | "warn";
}
```

## 3. Preserve the source of sensitivity

A single `sensitive?: boolean` cannot distinguish:

* Automatically sensitive because the key is currently `Authorization`
* Manually marked sensitive by the user
* Previously automatic, but the key was renamed
* A configured API-key row generated by the pipeline



Use either:

```ts
interface KeyValueRow {
  // ...
  sensitive: boolean;      // explicit user choice
}
```

with automatic sensitivity computed separately:

```ts
const isSensitive =
  row.sensitive ||
  isSensitiveHeaderName(row.key) ||
  row.origin === "configuredAuth";
```

Or store:

```ts
sensitivity: "none" | "user" | "auth";
```

The first approach is simpler. Renaming `Authorization` to `X-Custom` would remove automatic sensitivity but preserve a user-selected sensitive flag.

## 4. Add Rust-side payload validation and request limits

The frontend owns request preparation, but the Rust command should still treat IPC input as untrusted. The current command contract does not state backend limits for outbound request payloads. 

Before creating the reqwest request, Rust should validate:

```text
- executionId and tabId format
- method syntax or supported-method allowlist
- URL scheme is http or https
- header names and values are valid
- timeout is within a bounded range
- maxBodyBytes is within an application-defined range
- request body does not exceed an IPC request limit
- header count and aggregate header size are bounded
```

Suggested v0.1 limits:

```text
Raw/JSON request body: 10 MB
Headers: 200 rows
Aggregate request-header data: 256 KB
Timeout: 1 ms to 10 minutes
Response cap: 1 MB to 100 MB configurable
```

Return typed validation errors instead of allowing arbitrary reqwest builder errors.

This also prevents a damaged settings file from requesting an extremely large response allocation.

## 5. Add a global maintenance state for import replacement

The frontend import preconditions are good, and the backend rejects concurrent storage mutation. However, a new send could begin after the preconditions and append history while the live library is being replaced. A debounced write could also be scheduled after the initial flush. 

Add a backend maintenance state:

```rust
enum AppMode {
    Normal,
    ImportStaging,
    ImportApplying,
    Recovery,
}
```

While applying or recovering:

```text
- reject new storage mutations
- reject new sends
- pause history appends
- reject environment/settings/UI-state writes
- permit only import cancellation when safe
- expose a typed "maintenanceInProgress" error
```

The frontend should disable request sending and storage controls before calling `apply_workspace_import`.

Also clean up abandoned staged imports that have no active transaction marker—for example, staging directories older than 24 hours.

## 6. Define text-versus-binary response detection

The plan says non-UTF-8 responses are binary, but it does not fully define how `Content-Type` affects classification. 

A deterministic v0.1 rule:

```text
1. If Content-Type is textual:
   text/*, application/json, application/*+json,
   application/xml, application/*+xml,
   application/javascript, application/graphql
   → attempt UTF-8 decoding
   → invalid UTF-8 means binary

2. If Content-Type is clearly binary:
   image/*, audio/*, video/*, application/octet-stream,
   archives, PDFs, fonts
   → binary even when bytes happen to be valid UTF-8

3. Missing or unknown Content-Type:
   → valid UTF-8 with no NUL bytes = text
   → otherwise binary
```

Since charset conversion is intentionally excluded, display a message such as:

```text
This response is not valid UTF-8. Save it to inspect the original bytes.
```

## Smaller clarifications

### Simplify history URL handling

The history rule refers to storing a resolved URL for scratch requests “with no template,” but every scratch request still has a draft `RequestUrl`. 

Use one rule for all requests:

```text
Serialize the draft RequestUrl before variable substitution.
Preserve {{variable}} placeholders.
Redact every literal enabled query value.
```

This removes the special case entirely.

### Define deletion behavior for open tabs

When a saved request or containing folder is deleted:

```text
- prompt for confirmation
- tabs with dirty drafts remain open as unsaved scratch tabs
- clean tabs close, or become scratch tabs according to one consistent rule
- retained responses are released
- persisted tab IDs are removed from ui-state
```

Converting affected tabs into scratch tabs is safer because it avoids losing unsaved edits.

### Clarify atomic replacement implementation

`NamedTempFile::persist` behavior should be verified on both target platforms when replacing an existing file. Keep the storage API behind one helper so a Windows-specific replacement strategy can be added without changing callers. 

### Adjust one testing claim

A unit test cannot directly prove that data physically reached stable storage after `sync_all()`. It can verify that the synchronization method is invoked through an abstraction and test recovery after simulated failures. Avoid describing the test as proving physical durability.

## Final assessment

There are no remaining architecture blockers. The most important patches are:

1. Define the encoded representation of query rows.
2. Add the `prepareRequest` option matrix.
3. Validate and cap IPC request payloads in Rust.
4. Lock the whole application during import replacement.
5. Specify text-versus-binary response detection.

After those changes, further plan revisions are unlikely to add much value. M0 and M1 implementation should expose the next useful feedback.
